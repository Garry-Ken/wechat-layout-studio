# 主题驱动的知识库写作 Agent

这个产品的核心不是“把一篇文章排版好”，而是：

> 用户给一个主题，系统自动调取 Obsidian、飞书云文档和外部新信息，筛选合适知识、案例、观点和素材，重新排列组合，写成一篇有信息增量、有传播性、能服务主线业务的新文章。

## 输入

用户只需要提供：

- 主题：例如“为什么 AI 企业知识库会成为中小企业刚需”
- 目标读者：老板、销售负责人、交付负责人、知识管理负责人
- 业务主线：AI 企业知识库服务
- 写作目标：传播、获客、教育市场、案例沉淀、促成咨询
- 可选约束：文章长度、口吻、是否强 CTA、是否需要案例

## 核心流程

### 1. 主题理解

系统先把主题拆成几个问题：

- 这篇文章要解决读者什么问题？
- 读者为什么现在会关心？
- 和我的主线业务有什么关系？
- 文章需要证明什么核心判断？
- 哪些信息会增加可信度？
- 哪些内容只是背景噪音，应该排除？

输出一个“写作任务书”：

```json
{
  "theme": "AI 企业知识库为什么会成为中小企业刚需",
  "reader": "中小企业老板和业务负责人",
  "core_claim": "企业最先愿意付费的 AI，不是炫酷 Agent，而是能提升销售转化和交付效率的知识库系统。",
  "business_angle": "引导读者理解 AI 企业知识库服务的价值",
  "needed_evidence": ["行业趋势", "企业真实痛点", "可落地方法", "案例或场景"],
  "excluded": ["泛泛介绍 AI", "堆砌产品新闻", "没有业务落点的技术细节"]
}
```

### 2. 知识检索

不是把整个知识库丢给模型，而是先做检索规划。

检索来源：

- Obsidian：个人笔记、读书笔记、播客笔记、客户洞察、方法论
- 飞书云文档：项目文档、方案、客户会议纪要、销售话术、交付 SOP
- 外部素材：AI 产品更新、科技新闻、竞品动态、行业案例

检索策略：

- 先按主题关键词召回。
- 再按业务主线召回，例如“企业知识库、RAG、销售话术、交付效率、客户问题库”。
- 再按文章结构召回，例如“案例、反常识观点、方法清单、数据、类比”。

### 3. 素材取舍

每个候选素材都打分：

- 相关性：是否直接服务主题？
- 信息增量：读者看完是否获得新东西？
- 可信度：是否来自可靠来源或真实经验？
- 传播性：是否有反常识、金句、案例、清单？
- 获客价值：是否能自然指向 AI 企业知识库服务？
- 可表达性：是否容易转成公众号语言？

只有高分素材进入文章，不是召回多少用多少。

### 3.5 内容评分与校准

在真正写作前，先对候选角度做评分；初稿完成后，再对整篇文章做评分。评分机制参考 `cheat-on-content` 的闭环：

```text
打分 -> 盲预测 -> 发布 -> T+7d 复盘 -> 升级 rubric
```

公众号知识库文章的初始维度见 [内容评分与校准系统](./content-scoring-system.md)：

- TS：主题势能
- KA：知识库适配度
- IG：信息增量
- UV：读者可用性
- SC：传播/转发冲动
- CV：商业承接
- SN：结构叙事
- CE：可信证据

这一步的作用不是让 Agent 机械追分，而是让它明确：

- 这篇为什么值得写？
- 需要调用哪些知识？
- 哪些素材应该舍弃？
- 文章更偏传播还是更偏获客？
- 发布后要用什么数据验证判断？

### 4. 观点生成

系统不能只是摘要资料，而要组合出新观点：

- 从多个素材中抽象共同模式。
- 找出读者容易误解的地方。
- 提炼一个可传播的核心判断。
- 把你的业务经验变成方法论。

例如：

```text
不是企业缺 AI 工具，而是缺一个能让经验复用的知识底座。
AI 企业知识库的价值，不在“存资料”，而在“缩短新人、销售、交付到达正确答案的路径”。
```

### 5. 文章编排

推荐结构：

```text
标题：反常识或明确收益
开头：一个最新现象或真实痛点
核心判断：一句话说清观点
第一部分：为什么现在发生变化
第二部分：企业真实痛点是什么
第三部分：你的知识库/飞书/Obsidian 里有哪些经验可以支撑
第四部分：给读者一套可执行框架
第五部分：和 AI 企业知识库服务的自然连接
结尾：轻 CTA，引导咨询或交流
```

### 6. 写作

写作模型要遵守：

- 不堆资料。
- 不写空泛 AI 大词。
- 每一节都要回答“这对读者有什么用”。
- 每个观点尽量配一个例子、场景或方法。
- 结尾要自然连接业务，不硬广。

### 7. 编辑自检

生成后再跑一轮编辑 Agent：

- 信息增量够不够？
- 是否像真人写的，而不是资料拼贴？
- 标题有没有传播性？
- 第一屏能不能留住读者？
- 有没有明确观点？
- 有没有案例和方法？
- 是否自然服务 AI 企业知识库获客？
- 有没有事实不确定或需要标注来源的地方？

### 8. 排版与发布

最后才进入当前工具已有的能力：

- 自动排版
- 手机预览
- 公众号兼容 HTML
- 创建公众号草稿
- 手机端确认
- 发布或预约发布

## 系统架构

```text
主题输入
  -> 主题理解 Agent
  -> 检索规划 Agent
  -> Obsidian / 飞书 / 外部趋势检索
  -> 素材评分与去重
  -> 候选角度评分
  -> 观点重组 Agent
  -> 文章大纲 Agent
  -> 写作 Agent
  -> 内容评分与改稿 Agent
  -> 编辑与传播性检查 Agent
  -> 排版 Renderer
  -> 微信草稿 Publisher
  -> 手机确认 Approval
```

## 最小 MVP

第一版不需要全自动发布，先做这 5 件事：

1. 输入主题。
2. 读取本地 Obsidian vault。
3. 读取指定飞书文档或知识库文件夹。
4. 自动生成文章初稿和引用素材列表。
5. 排版后复制或创建公众号草稿。

第二步 MVP 应增加：

6. 发布前保存评分与盲预测。
7. 发布后 T+7d 录入阅读、分享、收藏、咨询线索。
8. 根据复盘结果更新内容评分 rubric。

## 需要用户授权的部分

- Obsidian vault 本地路径。
- 飞书开放平台授权，读取指定文档空间。
- 微信公众号 appID/appsecret，只放服务端。
- 外部信息源列表，例如 RSS、YouTube 频道、重点网站、X 账号。

## 产品判断

这个产品的真正壁垒不是排版，而是：

- 是否理解作者的长期业务主线。
- 是否能从私人知识库中选出该用的素材。
- 是否能把素材变成新观点，而不是摘要。
- 是否能写出对目标客户有用、愿意转发、愿意咨询的内容。
- 是否能把发布链路自动化到“作者只做确认和视频内容”。
